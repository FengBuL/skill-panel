# Skill Panel 产品问题复盘

更新时间：2026-06-16

## 复盘范围

本文记录 Skill Panel 在类目、卡片、详情、Markdown、解析、桌面打包与迁移交付中暴露的问题，以及本轮修复方式。目标是为后续产品迭代留下可追溯的 PM 复盘材料。

## 问题与解决方式

| 编号 | 问题 | 影响 | 根因 | 解决方式 | 验证 |
| --- | --- | --- | --- | --- | --- |
| 1 | 默认类目无法删除、修改或追加，Skill 会出现多个失控类目 | 用户无法管理 Skill 的真实归属 | 旧模型只支持 `skillCategoryOverrides` 单选覆盖，无法表达多类目和自定义类目 | 新增 `skillCategoryAssignments` 多类目设置，保留旧字段迁移；右键菜单支持添加、删除默认类目 | 前端测试 89 条通过 |
| 2 | 新增类目无法更改颜色和 logo | 自定义类目无法形成稳定视觉识别 | 颜色、图标、名称只允许内置四类 | 新增 `customCategories`，颜色、图标、名称与内置类目共用一套持久化逻辑 | 前端构建与测试通过 |
| 3 | 类目颜色只在 hover 或选中时显示 | 用户难以快速识别类目 | 左侧导航常态背景过弱 | 导航项增加常驻色条和淡色背景，hover/active 保持更强反馈 | 视觉 QA 通过 |

| 6 | “恢复默认”按钮语义不清 | 用户不理解恢复对象 | 文案过短 | 文案改为“恢复默认排序”/“Restore default order” | 前端测试通过 |
| 7 | 卡片、列表、详情展示 CODEX/AGENTS/PLUGIN 等来源 | 信息对普通用户价值低，增加噪声 | 技术来源被直接暴露到用户界面 | 卡片、列表、详情去掉来源展示；后台仍保留来源用于只读保护 | 前端测试更新并通过 |
| 8 | 详情页“快速理解”和“Prompt 匹配测试”造成冗余和重叠 | 详情内容重复、阅读负担高 | 自动分析模块与正文预览并列，布局高度不足 | 删除这两个模块，仅保留 Skill 检查、描述、Markdown 正文和必要元信息 | 前端测试通过 |
| 9 | lark Skill 详情加载失败，提示 `Unsupported nested frontmatter line: requires:` | 飞书相关 Skill 无法查看 | Rust 解析器只支持简单 scalar 和数组，不支持嵌套 YAML | 解析器兼容一层嵌套对象和嵌套数组；lark Skill 标记为只读可查看 | Rust 37 条测试通过 |
| 10 | Markdown 编辑窗口和渲染窗口高度、边框不一致，滚动体验差 | 长 Skill 编辑时频繁滚动，左右视图难对齐 | 两栏组件使用不同高度和样式，未建立滚动联动 | 两栏统一高度、边框、阴影；编辑区滚动/点击/键盘后按相对位置同步预览 | 前端构建与视觉 QA 通过 |

| 14 | 桌面构建没有更新 | 用户打开的仍是旧 exe | 正在运行的旧 `skill-panel.exe` 锁定 release 文件 | 关闭旧进程后重新执行 Tauri release 构建 | `tauri build` 通过 |

## 产品层总结

1. 类目需要作为一等数据模型存在，标签只能做补充标记。否则用户看到的导航、分组、筛选会产生不一致。
2. 面向普通用户的界面不应暴露实现来源，技术来源保留在后台用于权限和安全判断即可。
3. Markdown 是 Skill 的核心内容，详情页应以标准预览和稳定编辑体验为中心，辅助模块要克制。
4. 本地桌面应用的更新需要处理进程锁定，打包前应检查旧进程是否仍在运行。
5. 复杂 Skill 文件要容忍真实世界的 frontmatter 差异，读取失败会直接破坏用户信任。
